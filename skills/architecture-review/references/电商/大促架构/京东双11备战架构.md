# 京东双11备战架构参考

## 核心设计原则

1. **流量分级** - 重点保护核心购买链路的资源
2. **兜底策略** - 任何依赖都要有降级方案
3. **保守限流** - 宁可误杀不能放过

## 关键架构决策

### 1. 多级缓存设计

```
用户请求
  ↓
CDN（静态资源）
  ↓
Nginx（入口限流）
  ↓
应用层Redis缓存（热点数据）
  ↓
数据库（最终一致）
```

**缓存失效策略**：
- 热点数据：写请求穿透后写缓存，设置TTL
- 非热点数据：读请求直接回源，不缓存

### 2. 下单与库存解耦

```
下单请求
  ↓
创建订单（DB） + 预占库存（Redis）
  ↓
异步消息队列
  ↓
库存服务确认
  ↓
订单状态变更
```

**为什么这样设计**：
- 下行业务单元测试可达性强
- 库存和下单可以独立扩容
- 库存可以接受延迟确认，但下单必须快速返回

### 3. 限流保护

| 层级 | 限流策略 | 阈值设置 |
|------|----------|----------|
| Nginx | 分布式限流 | 5万QPS |
| 应用层 | 单机限流 | 1000QPS |
| 数据库 | 连接池保护 | 最大200连接 |

**阈值设定原则**：
- 平时流量的 3-5 倍作为基准
- 大促前 1 周根据压测结果调整
- 设置预警值，到 80% 阈值就报警

## 常见风险点

| 风险 | 预案 |
|------|------|
| Redis 集群宕机 | 切换到本地缓存 + 禁止写入 |
| 数据库连接池打满 | 直接返回"系统繁忙" |
| 下单服务实例全部不可用 | 限流 + 返回友好错误 |
| 库存队列积压 | 降级为普通发货，不保证库存 |

## 经验教训

1. **全链路压测必须做** - 不能只测单个服务，要模拟真实下单链路
2. **限流要提前人工确认** - 自动限流可能误杀正常流量
3. **降级要有明确开关** - 出现问题时可以一键切换
4. **监控要覆盖到核心指标** - 下单转化率、库存预占成功率、支付成功率
