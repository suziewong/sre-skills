# 库存扣减防超卖方案

## 问题本质

```
10个库存
  ↓
100个并发请求同时读取
  ↓
都看到还有库存
  ↓
100个请求都扣减成功
  ↓
超卖90个！
```

## 方案一：数据库乐观锁（最常用）

### 实现

```sql
UPDATE inventory
SET stock = stock - :quantity,
    version = version + 1,
    updated_at = NOW()
WHERE product_id = :product_id
  AND stock >= :quantity
  AND version = :current_version
  AND deleted = 0;

-- affected_rows == 1 表示成功
-- affected_rows == 0 表示失败（库存不足或版本冲突）
```

### 优点
- 实现简单，不依赖外部组件
- 性能好，无锁等待

### 缺点
- 高并发下失败率较高
- 库存充足时用户体验差（大量失败重试）

---

## 方案二：Redis 预扣 + 数据库最终一致

### 实现

```
下单请求
  ↓
Redis DECR（原子扣减）
  ↓
成功 → 异步写数据库
失败 → 库存不足，直接返回
```

```python
# Lua 脚本保证原子性
LUA_DECR_STOCK = """
local stock = redis.call('GET', KEYS[1])
if stock and tonumber(stock) >= tonumber(ARGV[1]) then
    return redis.call('DECRBY', KEYS[1], ARGV[1])
else
    return -1
end
"""

result = redis.eval(LUA_DECR_STOCK, 1, f'inventory:{product_id}', quantity)
if result == -1:
    raise OutOfStockError()
```

### 优点
- 性能极高，支持百万QPS
- 用户体验好，快速失败

### 缺点
- Redis 故障时需要降级方案
- 需要补偿机制处理 Redis 和 DB 不一致

## 方案三：数据库悲观锁（不推荐）

```sql
BEGIN;
SELECT * FROM inventory WHERE product_id = ? FOR UPDATE;
-- 扣减库存
UPDATE inventory SET stock = stock - ? WHERE id = ?;
COMMIT;
```

**问题**：严重性能问题，高并发下数据库连接池会被打满

---

## 推荐方案：多级库存 + Redis 预扣

```
用户层
  ↓
Redis 库存（热点商品，实时扣减）
  ↓ [Redis 失败或非热点商品]
DB 库存（兜底，最终一致）
```

### 完整实现逻辑

```python
def扣减库存(product_id, quantity):
    # 第一层：Redis 乐观扣减
    key = f"inventory:{product_id}"
    new_stock = redis.decrby(key, quantity)

    if new_stock >= 0:
        # 成功，异步落库
        send_to_mq("inventory_deduct", {
            "product_id": product_id,
            "quantity": quantity,
            "source": "redis"
        })
        return Success()

    # 第二层：Redis 库存不足，回补并尝试数据库
    redis.incrby(key, quantity)  # 回补Redis
    return deduct_from_db(product_id, quantity)

def deduct_from_db(product_id, quantity):
    # 数据库乐观锁重试（最多3次）
    for i in range(3):
        result = db.execute("""
            UPDATE inventory
            SET stock = stock - %s,
                version = version + 1
            WHERE product_id = %s
              AND stock >= %s
              AND version = %s
        """, [quantity, product_id, quantity, current_version])

        if result.affected_rows == 1:
            return Success()

        # 失败重试前先查一下当前库存
        current = db.query("SELECT stock FROM inventory WHERE product_id = %s", product_id)
        if current.stock < quantity:
            return OutOfStock()

    return TryLater()  # 3次失败，服务繁忙
```

## 库存扣减的坑

| 坑 | 后果 | 解决方案 |
|----|------|----------|
| 浮点数 | 超卖 | 用整数或Decimal |
| 负库存 | 超卖 | WHERE stock >= quantity |
| 超时未释放 | 库存占用 | 订单超时自动释放 |
| 重复扣减 | 多扣 | 幂等键 |
| 缓存击穿 | 服务雪崩 | 限流 + 多级缓存 |

## 库存回滚场景

| 场景 | 处理方式 |
|------|----------|
| 订单超时未支付 | 定时任务扫描，释放库存 |
| 用户取消订单 | 即时回滚 Redis + DB |
| 支付失败 | 即时回滚 |
| 商家取消订单 | 即时回滚 + 补偿消息 |

```python
def 回滚库存(product_id, quantity, order_id):
    # Redis 回滚
    redis.incrby(f"inventory:{product_id}", quantity)

    # DB 回滚（乐观锁）
    db.execute("""
        UPDATE inventory
        SET stock = stock + %s,
            version = version + 1
        WHERE product_id = %s
    """, [quantity, product_id])

    # 记录流水（对账用）
    db.execute("""
        INSERT INTO inventory_log
        (product_id, quantity, type, order_id, created_at)
        VALUES (%s, %s, 'ROLLBACK', %s, NOW())
    """, [product_id, quantity, order_id])
```

## 库存热点问题

大促时单个爆款商品可能 QPS 超过 10 万，解决方案：

1. **库存分桶**：将库存分成 N 份，每份放到不同的 Redis 节点
2. **异步化**：下单不实时扣库存，改为预占 + 延迟确认
3. **限流**：热点商品入口限流，保护下游
