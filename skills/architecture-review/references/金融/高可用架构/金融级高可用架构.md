# 金融级高可用架构参考

## 金融系统的特殊性

| 维度 | 互联网系统 | 金融系统 |
|------|-----------|----------|
| 数据准确性 | 99%就行 | 必须100% |
| 可用性 | 99.9% | 99.999% |
| 一致性 | 最终一致 | 强一致 |
| 容灾 | 可丢数据 | 数据零丢失 |
| 合规 | 无强制要求 | 监管强制要求 |

## 核心架构原则

### 1. 多地多活

```
华北机房（主）
  ├── 应用集群（至少3节点）
  ├── MySQL主库
  ├── Redis主库
  └── Kafka主集群

华东机房（从）
  ├── 应用集群（至少3节点）
  ├── MySQL从库（实时同步）
  ├── Redis从库
  └── Kafka从集群

华南机房（灾备）
  └── 数据异步复制
```

**RPO（恢复点目标）**：≤ 1秒
**RTO（恢复时间目标）**：≤ 30秒

### 2. 强一致性保障

#### 分布式事务方案对比

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|----------|
| 2PC | 强一致 | 低 | 高 | 同库分表 |
| TCC | 最终一致 | 高 | 高 | 跨服务 |
| Saga | 最终一致 | 高 | 中 | 长事务 |
| 本地消息表 | 最终一致 | 中 | 中 | 异步场景 |

**金融系统推荐**：核心交易用 TCC 或 2PC，非核心用 Saga

### 3. 数据安全多层防护

```
应用层
  ├── 输入校验（防注入）
  ├── 金额精度处理（Decimal）
  └── 操作审计日志

服务层
  ├── 连接池隔离（读/写分离）
  ├── 超时控制（合理阈值）
  └── 熔断降级

数据库层
  ├── 主从复制（半同步）
  ├── 定期全量备份 + 增量备份
  └── 敏感数据加密
```

## 核心交易架构示例

### 支付网关架构

```
外部请求
    ↓
接入层（SSL卸载、限流）
    ↓
交易路由（根据商户/通道动态路由）
    ↓
风控前置（规则引擎 + 模型）
    ↓
交易核心（强一致性）
    ├── 账户服务（余额扣减）
    ├── 订单服务（状态管理）
    └── 通道服务（上游对接）
    ↓
异步通知（消息队列 + 重试）
```

### 账户服务设计要点

```python
# 错误的做法：浮点数
balance = 100.50  # 浮点数精度问题！

# 正确的做法：整数（分）
balance = 10050  # 单位：分

# 或使用 Decimal
from decimal import Decimal
balance = Decimal('100.50')
```

```sql
-- 余额扣减必须带版本号乐观锁
UPDATE account
SET balance = balance - :amount,
    version = version + 1
WHERE user_id = :user_id
  AND balance >= :amount
  AND version = :current_version

-- 检查影响行数
IF affected_rows != 1:
    raise ConcurrencyError()
```

## 容灾设计

### 1. 切换决策树

```
故障检测
  ↓
是否核心链路？
  ├── 否 → 降级（非核心功能）
  └── 是 → 是否跨区域？
           ├── 否 → 同城切换
           │     └── 切换到备用节点
           └── 是 → 跨区域切换
                 └── 启动异地灾备
```

### 2. 切换时间要求

| 场景 | RTO | RPO |
|------|-----|-----|
| 同城切换 | < 30秒 | 0 |
| 跨城切换 | < 5分钟 | < 1秒 |
| 异地灾备 | < 30分钟 | < 1小时 |

### 3. 数据同步方案

**实时同步**：半同步复制
```sql
-- MySQL 半同步复制配置
SET GLOBAL rpl_semi_sync_master_wait_point = AFTER_SYNC;
```

**异步复制**：异地机房延迟 < 1秒

**数据校验**：每日核对总账，确保主从数据一致
