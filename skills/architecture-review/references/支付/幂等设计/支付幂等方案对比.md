# 支付幂等方案对比

## 为什么支付必须幂等

```
用户点击支付
  ↓
网络超时/响应慢
  ↓
用户再次点击
  ↓
如果没幂等 → 重复扣款！
```

## 方案一：唯一键 + 防重表（推荐）

### 实现方式

```sql
-- 防重表
CREATE TABLE payment_idempotent (
    idempotent_key VARCHAR(64) PRIMARY KEY,  -- 业务方生成的唯一键
    payment_status VARCHAR(32),              -- SUCCESS/FAIL/PENDING
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    INDEX idx_status (payment_status)
);

-- 支付逻辑
BEGIN TRANSACTION;
  SELECT * FROM payment_idempotent WHERE idempotent_key = ? FOR UPDATE;
  -- 如果已存在，根据状态返回
  -- 如果不存在，插入并执行支付
COMMIT;
```

### 优点
- 实现简单，逻辑清晰
- 数据库原生支持，不依赖外部组件
- 可追溯，查询方便

### 缺点
- 依赖数据库，高并发下数据库是瓶颈
- 唯一键冲突时需要业务方重试

## 方案二：Redis + 数据库双写

### 实现方式

```python
# 生成幂等键
idempotent_key = f"payment:{user_id}:{order_id}:{timestamp}"

# 先写Redis（快速失败）
if redis.setnx(idempotent_key, "PROCESSING", ex=300):
    # 获取分布式锁成功，执行支付
    result = do_payment(order)
    if result.success:
        redis.set(idempotent_key, "SUCCESS", ex=86400)
        db.save(payment_record)
    else:
        redis.delete(idempotent_key)
else:
    # 已在处理中或有记录，查询状态
    status = redis.get(idempotent_key)
```

### 优点
- Redis 性能好，高并发下吞吐能力强
- 天然支持过期自动清理

### 缺点
- 引入 Redis 依赖，增加复杂度
- Redis 和 DB 数据可能短暂不一致

## 方案三：消息队列 + 本地消息表（推荐用于高并发）

### 实现方式

```
1. 本地消息表记录支付请求
2. 发送消息到 MQ
3. 消费消息执行支付
4. 更新本地消息表状态
5. 定时扫描重试失败消息
```

### 优点
- 高吞吐量，适合高并发场景
-天然幂等，消息可重试消费

### 缺点
- 实现复杂度高
- 延迟略高（异步）

## 方案对比

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|----------|
| 防重表 | 强 | 中 | 低 | 中低并发 |
| Redis+DB | 弱（最终一致） | 高 | 中 | 高并发 |
| MQ+本地消息表 | 最终一致 | 最高 | 高 | 超高并发 |

## 幂等键设计原则

```
幂等键 = 业务标识 + 操作类型 + 时间戳 + 随机数

示例：
payment:u12345:o98765:20241121:abc123

组成部分：
- payment: 操作类型
- u12345: 用户ID
- o98765: 订单ID
- 20241121: 日期（可选，用于分区）
- abc123: 随机后缀（可选，保证唯一性）
```

## 常见错误

1. **幂等键不包含用户ID** - 导致不同用户同一订单可以重复支付
2. **只靠数据库唯一索引** - 高并发下可能先插成功再回滚
3. **不处理超时** - 超时时要主动查询状态，不能直接失败
4. **Redis 锁不设置过期时间** - 可能死锁

## 支付状态机

```
PENDING（待支付）
    ↓ [用户支付成功]
SUCCESS（支付成功）
    ↓ [退款申请]
REFUNDING（退款中）
    ↓ [退款成功]
REFUNDED（已退款）
    ↓ [退款失败]
REFUND_FAILED（退款失败）
```

任何状态变更都要先查当前状态，防止状态回退。
