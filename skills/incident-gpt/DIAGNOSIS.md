# Incident-GPT 诊断流程

## 核心流程

```
┌─────────────────────────────────────────────────────────────┐
│                      故障分析流程                            │
└─────────────────────────────────────────────────────────────┘

   ┌──────────────┐
   │  原始数据输入  │  日志 / 错误堆栈 / Trace / 监控指标
   └──────┬───────┘
          │
          ▼
   ┌──────────────┐
   │  数据解析      │  提取时间戳、错误类型、服务名
   └──────┬───────┘
          │
          ▼
   ┌──────────────┐
   │  事件提取      │  识别关键事件节点
   └──────┬───────┘
          │
          ▼
   ┌──────────────┐
   │  时间线构建    │  按时间排序，生成时间线
   └──────┬───────┘
          │
          ▼
   ┌──────────────┐
   │  根因推断      │  基于错误模式分析
   └──────┬───────┘
          │
          ▼
   ┌──────────────┐
   │  影响评估      │  用户数/请求量/SLO
   └──────┬───────┘
          │
          ▼
   ┌──────────────┐
   │  RCA 报告生成  │  Markdown 格式输出
   └──────────────┘
```

## LLM Prompt 模板

### Step 1: 事件提取

```
你是一个 SRE 故障分析专家。请从以下日志中提取关键事件节点。

日志内容：
{{logs}}

请提取：
1. 每个关键事件（时间、事件描述、涉及服务）
2. 异常模式的开始和结束时间
3. 涉及的错误类型

输出格式：JSON
```

### Step 2: 根因分析

```
基于以下故障信息，进行根因分析：

故障时间线：
{{timeline}}

错误日志摘要：
{{error_summary}}

请分析：
1. 直接原因（Immediate Cause）
2. 根本原因（Root Cause）
3. 为什么这个问题没有在发生前被阻止
4. 为什么这个问题导致了业务影响

输出格式：结构化文本
```

### Step 3: RCA 报告生成

```
请根据以下信息，生成完整的 RCA 报告：

- 故障描述：{{description}}
- 时间线：{{timeline}}
- 根因：{{root_cause}}
- 影响评估：{{impact}}

请生成包含以下部分的完整报告：
1. 基本信息（时间、影响、持续时长）
2. 详细时间线
3. 根因分析
4. Action Items（分类：预防/止损/复盘）
5. 经验教训

格式：Markdown
```

## 常见错误模式匹配

| 错误模式 | 可能根因 | 检查方向 |
|----------|----------|----------|
| connection pool full | 数据库连接池耗尽 | 慢查询、连接泄漏 |
| timeout | 网络或服务超时 | DNS、服务发现、网络抖动 |
| OOM / OutOfMemory | 内存溢出 | 内存泄漏、大对象 |
| 429 / rate limit | 请求限流 | 流量突增、爬虫 |
| 503 / unavailable | 服务不可用 | 依赖服务挂了、主动熔断 |
| disk full | 磁盘空间不足 | 日志文件、临时文件 |
| CPU 100% | CPU 打满 | 死循环、GC 问题 |

## 数据质量检查

输入数据如果满足以下条件，分析质量更高：

- [ ] 包含时间戳
- [ ] 包含错误堆栈或错误码
- [ ] 能识别出涉及的服务名
- [ ] 有请求量/RT 等监控指标参考

如果数据不足，AI 会基于现有信息给出最佳推测，并在报告中注明"数据可能不完整"。
